## 2026-05-17 - Avoid automatic exposure of profiling and metrics endpoints on DefaultServeMux
**Vulnerability:** Profiling (`/debug/pprof/*`) and metrics (`/debug/vars`) endpoints were automatically exposed on the globally accessible `http.DefaultServeMux` due to anonymous imports of `net/http/pprof` and `expvar` in `cmd/tesseract/gcp/main.go` and `cmd/tesseract/posix/main.go`. This could lead to sensitive information disclosure (CWE-200), as external attackers or unauthorized internal actors might access execution profiles, memory usage, and BadgerDB metrics.
**Learning:** Anonymous imports of `net/http/pprof` and `expvar` automatically register their respective HTTP handlers onto `http.DefaultServeMux` during package initialization. If a public-facing application uses `http.DefaultServeMux` (directly or indirectly via `http.ListenAndServe(addr, nil)` or passing a multiplexer that includes it), these debugging endpoints are exposed to the public. In cloud environments where `http.Server` entry points are used (like `posix` and `gcp`), relying on other observability tools (like OpenTelemetry) while accidentally leaving these imports in place creates an unnecessary security risk.
**Prevention:**
- Do not use anonymous imports for `net/http/pprof` or `expvar` in production entry points unless intentionally exposing them on an internal-only port.
- Use explicit multiplexers (e.g., `http.NewServeMux()`) instead of `http.DefaultServeMux` to ensure only intended endpoints are registered and exposed.
- For profiling and metrics, prefer dedicated, secure telemetry exporters (like OpenTelemetry) as implemented elsewhere in the cloud personalities.
